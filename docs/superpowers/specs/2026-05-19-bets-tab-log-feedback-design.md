# Bets-tab Log/Place feedback — design

## Review Pack

**What we're building** — Replace the full-page `location.reload()` that fires after a successful Log or Place click on the MLB bets tab with an in-place DOM swap, mirroring how the parlay tab already handles `logParlay`/`placeParlay`. After clicking Log or Place, the card's button area updates immediately to a "placed" label and a success toast appears — no jarring page refresh.

**Key decisions**

1. **Generalize the parlay tab's `_replaceActionCell` helper rather than write a new one.** Alternative: keep `_replaceActionCell` parlay-only and add a separate `_replaceBetsActionCell`. Rejected because the only difference is the walk-up target (`.actions` on bets-tab V8 cards vs `.rt-td` on parlay reactable rows) — one-line generalization is cheaper than two near-identical helpers.
2. **Fire immediately on click (no confirmation modal).** Alternative: parlay-style stake modal. Rejected because the bets tab already has an inline editable Risk field (PR C, 2026-05) — the stake editor exists on the card itself, making a popup redundant and adding an extra click for no UX win.
3. **JS-built label HTML matches what R renders for `status='placed'` rows on next dashboard regen.** Both produce `<span class="placed-bet-label" data-*="...">placed</span>` (or `placed · #<ticket>` when Place returns a ticket). No visual discontinuity between "just placed via JS" and "logged + regenerated by R."
4. **Scope = bets-tab `logBet` + `placeBet` only.** Retry-on-error path (red pill → Retry button) and the trifecta tab are explicitly out of scope. Same reload pattern applies, but they're rare paths and the user hasn't complained about them.

**Risks / push back here**

- **Data-attribute forwarding.** The new `<span>` carries forward `data-hash`, `data-edge`, `data-size`, etc. so any post-render filters (e.g. by edge / bookmaker) still classify the row correctly after the swap. If a future filter looks at `[data-fill-status]` specifically and treats `not_placed` differently, the JS-side swap will lag the R-rendered version by one regen cycle. Mitigation: forward `data-fill-status="placed"` on the swap.
- **Click-through prevention.** Currently `btn.disabled = true; btn.textContent = 'Logging...'` runs immediately on click. We keep that. After successful POST and DOM swap, the button no longer exists — so the disabled flag becomes moot. No race-condition risk.
- **`placed-bet-label` styling assumption.** The CSS class is defined and used today (R renders it at `mlb_dashboard.R:1614`). We're assuming it renders identically inside `.actions` as it does at the wider card scope. Worth eyeballing in browser, but no structural reason it wouldn't.

**Worth understanding** (opt-in)

1. **`Element.closest(selector)`** — DOM API that walks up the tree from the element until it finds an ancestor matching the selector. Like `getElementById` but searching upward instead of globally. R analogue is roughly `Find()` walking a list of parents. Used in `_replaceActionCell` to locate the cell wrapper without hardcoding the DOM depth.
2. **Why a static HTML file makes reload useless here**: `report.html` is generated once by R, then Flask serves the same bytes on every request until the next `Rscript mlb_dashboard.R`. So `location.reload()` re-fetches the same content. JS DOM mutation is the only way to reflect new state until the next R regen.

## Design

### Files touched

- `Answer Keys/MLB Dashboard/mlb_dashboard.R` — JS string updates only. No R logic changes.

### Component 1: Generalize `_replaceActionCell`

Located at `mlb_dashboard.R` ~line 5006. Current implementation walks up to `.rt-td` (Reactable cell wrapper, used by the parlay table).

**Change**: add `.closest('.actions')` ahead of the `.rt-td` check so the same helper finds the V8 card's `<div class="actions">` wrapper inside `.bet-card-v8 .hero`.

```javascript
function _replaceActionCell(btn, node) {
  var cell = btn.closest('.actions') || btn.closest('.rt-td') || btn.parentNode;
  while (cell.firstChild) cell.removeChild(cell.firstChild);
  cell.appendChild(node);
}
```

The bets-tab cards always have a `.actions` ancestor; the parlay reactable cells do not. The fallthrough preserves existing parlay behavior verbatim.

### Component 2: Rewrite `logBet` success path

Located at ~line 4218. Current behavior on `result.success !== false`:

```javascript
setTimeout(function() { location.reload(); }, 400);
```

**Replace with**: build a `<span class="placed-bet-label">` carrying the button's `data-*` attributes, swap into the action cell, show a success toast.

```javascript
// success path
var span = document.createElement('span');
span.className = 'placed-bet-label';
span.textContent = 'placed';
// Forward all data attrs so post-render filters still classify the row.
for (var key in btn.dataset) { span.dataset[key] = btn.dataset[key]; }
span.dataset.fillStatus = 'placed';
_replaceActionCell(btn, span);
showToast('Logged $' + Math.round(parseFloat(body.actual_size)), 'success');
```

No reload. Error paths unchanged (still show error toast, restore button label).

### Component 3: Rewrite `placeBet` success path

Located at ~line 4150 (the `result.status === 'placed'` branch). Current:

```javascript
showToast('Placed at ' + book + ticket, 'success');
setTimeout(function() { location.reload(); }, 800);
```

**Keep** the existing toast verbatim. **Replace** the reload with the same DOM swap. Label text differs by whether a ticket was returned:

```javascript
var ticket = result.ticket_number;
var labelText = ticket ? ('placed · #' + ticket) : 'placed';
var span = document.createElement('span');
span.className = 'placed-bet-label';
span.textContent = labelText;
for (var key in btn.dataset) { span.dataset[key] = btn.dataset[key]; }
span.dataset.fillStatus = 'placed';
_replaceActionCell(btn, span);
showToast('Placed at ' + book + (ticket ? ' #' + ticket : ''), 'success');
// no setTimeout / location.reload
```

Other branches in `placeBet` (`playwright_launched`, `price_moved`, error) keep their existing behavior — only the `placed` branch loses the reload.

### Data flow

```
Click Log → button.disabled=true, textContent='Logging...'
         → fetch POST /api/log-bet
         → server INSERT into placed_bets with status='placed'
         → response {success: true, status: 'placed', ticket_number: null}
         → JS builds <span class="placed-bet-label"> with forwarded data-*
         → _replaceActionCell walks up to .actions, replaces children
         → showToast('Logged $X', 'success')
```

Identical structure for `placeBet`, except the server flow goes through `single_placer.place_single` (or Playwright) and the response may include a `ticket_number`.

### Error handling

Unchanged. If `result.success === false` or fetch rejects:
- error toast
- button restored to `'Log'` / `'Place Bet'` text
- `btn.disabled = false`

No DOM swap on failure. The user can immediately retry.

### Testing

Manual click-and-verify on the running dashboard at `http://localhost:8083`:

1. **Log on a non-WZ bet card** (e.g. Pinnacle, DraftKings). Confirm:
   - Buttons disappear, replaced by `placed` label
   - Toast appears: `Logged $<size>`
   - URL bar does not flash / reload
   - Network tab in dev tools shows only the POST, no document reload
   - Row inserted into `placed_bets` with `status='placed'` (verify via DuckDB)
2. **Place on a WZ bet card** with a valid account selected. Confirm:
   - Same DOM swap behavior
   - Toast still shows `Placed at wagerzon #<ticket>`
   - Label includes `placed · #<ticket>`
3. **Failure cases** still surface as toasts without DOM swap:
   - Log on a hash the server rejects (force via DevTools modifying `data-hash`)
   - Network offline → catch branch → error toast
